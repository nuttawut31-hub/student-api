# ขั้นตอนที่ 3.2 ทบทวนคุณภาพของโค้ด (Peer Review)

## 67160028 ณัฐวุฒิ รู้การนา

### 1) โค้ดใน `index.js` มีการตรวจสอบข้อมูลนำเข้า (validation) หรือไม่?
* **มีเบื้องต้น:** ตรวจสอบความครบถ้วนของฟิลด์ข้อมูลในคำขอ POST และ PUT (หากข้อมูลไม่ครบจะคืน `400 Bad Request`)
* **สิ่งที่ควรเพิ่ม:** ควรตรวจสอบประเภทข้อมูล (Data Type เช่น `id` ต้องเป็นตัวเลข) และการจำกัดรูปแบบความยาวข้อความเพิ่มเติม

### 2) ทุก route มีการตอบกลับด้วย Status Code ที่เหมาะสมหรือไม่?
* **เหมาะสมตามมาตรฐาน REST API:**
  * **`200 OK`**: ใช้ดึงข้อมูลสำเร็จ, แก้ไขสำเร็จ (PUT) และลบสำเร็จ (DELETE)
  * **`201 Created`**: ใช้เมื่อเพิ่มข้อมูลใหม่สำเร็จ (POST)
  * **`400 Bad Request`**: ใช้เมื่อ Client ส่งข้อมูลเข้ามาไม่ครบถ้วน
  * **`404 Not Found`**: ใช้เมื่อไม่พบ ID ของนักศึกษาหรือรายวิชาในระบบ

### 3) โครงสร้างโค้ดในไฟล์เดียวยังอ่านง่ายอยู่หรือไม่ และจะเกิดปัญหาใดหากมี route เพิ่มขึ้นอีก 10 เส้นทาง?
* **หากใช้ไฟล์เดียว:** จะเกิดปัญหาโค้ดปนเปกันขนาดใหญ่ (Spaghetti Code) ทำให้อ่าน/แก้ไขยาก เกิดความขัดแย้งของโค้ด (Merge Conflict) ได้ง่ายเมื่อทำงานเป็นทีม และทดสอบระบบได้ยาก
* **แนวทางแก้ไขปัจจุบัน:** ใช้ `express.Router()` แยกโมดูลออกมาไว้ที่ `routes/student.js` ทำให้โค้ดเป็นระบบ ปรับปรุงง่าย และขยายตัวในอนาคตได้ดีกว่า
